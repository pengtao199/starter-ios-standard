# Performance Refactor Rules

## Scope

适用于性能优化、结构重构、状态收敛、启动优化和 SwiftUI 渲染问题排查。

## Goal

优化必须有现象、链路、根因和验证点，避免用大改掩盖小问题。

## Rules

- 开发、优化、性能任务先给出：现象 -> 链路 -> 根因 -> 验证点。
- 优先定位最小问题边界，再改最小代码面。
- 保持用户可见行为不变，除非任务明确要求改交互。
- 重构前先确认命中的专项规则，尤其是分层、状态、本地化和原生 UI。
- SwiftUI 性能问题先检查状态粒度、重复计算、列表 identity、同步阻塞和视图拆分。
- 启动性能问题先检查组合根、同步 IO、网络等待、订阅刷新和重型初始化。
- 不确定或争议强的 iOS 能力先查 Apple 官方文档。
- 涉及公共组件或基础层时，验证至少覆盖一个调用方。

## Avoid

- 没有复现或验证点就做大范围重构。
- 把性能问题归因到 SwiftUI 本身而不检查状态流。
- 顺手改命名、格式、架构边界等无关内容。
- 为未来性能提前引入缓存、下载、媒体或复杂并发能力。

## Checklist

- 是否写清现象、链路、根因、验证点。
- 是否只改了当前问题需要的范围。
- 是否保护了已有本地化、合规和配置边界。
- 是否说明已运行或无法运行的验证。
